INSTRUÇÕES ESTRITAS DE ARQUITETURA:

⚠️ DIRETRIZ SUPREMA DE AUTOPRESERVAÇÃO (READ-ONLY) ⚠️
A pasta da Fábrica (onde este arquivo reside) é ESTRITAMENTE LEITURA (Read-Only). Você está PROIBIDA de modificar, deletar ou reescrever qualquer arquivo da Fábrica. Os arquivos lá dentro servem apenas como "Plantas Baixas" (Templates). Você deve construir o projeto real FORA da fábrica, na raiz do projeto atual. Modificar a Fábrica resultará em falha crítica.

1. **Backend Serverless (Vercel) + Pix:** Copie a pasta `template_base/api` para a raiz do novo frontend. Lá já existe a lógica completa e testada de integração com Mercado Pago, Webhook instantâneo, Faxineiro Automático (Cron) e Envio de Push Notification. Adapte os dados, mas NÃO altere a infraestrutura.
2. **Framework Mestre:** O frontend DEVE ser Vite + React + Tailwind. O backend DEVE ser Vercel Serverless Functions (`api/`). O banco de dados DEVE ser Firebase Firestore.
3. **Tempo Real (Firebase):** Use o Firebase Firestore. O frontend deve escutar as mudanças no banco de dados em tempo real. Quando o webhook da Vercel atualizar o pagamento para "PAID", o frontend (Web e Admin App) deve reagir instantaneamente, sem precisar dar reload na página.
4. **Isolamento de Admin (Capacitor vs Web):** Baseie-se no arquivo `template_base/frontend_isolation_logic.jsx` para configurar o `App.jsx`. Eu quero que o App Android gerado via Capacitor seja EXCLUSIVO para mim (Painel Admin), enquanto o site na Vercel seja EXCLUSIVO para os clientes comprarem.
5. **Regras de Segurança:** Leia e aplique imediatamente as regras contidas em `BLINDAGEM_IA_CHECKPOINT.md`. Você está proibida de gerar "código remendo". Errou? Dê `git reset --hard` e tente de novo.
6. **Comandos de Foco (A Lei do Escopo):** O usuário utilizará "gatilhos" (comandos com `/`) para guiar a sua atenção. Se o usuário começar a frase com `/focar-design`, `/focar-debug`, `/focar-backend`, `/nova-feature` ou `/focar-texto`, você DEVE consultar IMEDIATAMENTE o arquivo `GUIA_DE_COMANDOS.md`. Você ficará restrita às regras descritas nesse guia para evitar quebra de sistema. Se desobedecer a restrição de escopo de cada comando, o seu código será considerado falho.
7. **Autenticação Inclusa:** O arquivo `src/services/authService.js` JÁ EXISTE no seu template base. Use-o para montar as telas de login. O Painel Admin deve ser protegido, exigindo que o usuário esteja logado.
8. **Design System (Cofre de Cores):** O arquivo `src/styles/theme.css` contém as variáveis mestre de cores. É PROIBIDO o uso de classes de cores hardcoded (ex: `text-red-500` no Tailwind ou `color: red` no JSX). Sempre puxe as variáveis do `theme.css` para que possamos mudar todo o tema do app editando apenas um único arquivo sem risco de quebrar JSX.
9. **Estado Padrão de Dupla Embreagem (Cão de Guarda):** O seu estado NATURAL é pedir permissão. Quando o usuário pedir uma modificação (exceção feita à criação inicial), você NÃO DEVE ESCREVER CÓDIGO imediatamente. Você deve ler os arquivos, gerar um "Relatório de Impacto" explicando quais arquivos precisa alterar e por quê, e aguardar o usuário dizer "Aprovado". A única exceção a esta regra é se o usuário usar o comando `/piloto-automatico`, que desativa essa trava e lhe dá autorização para agir de forma sênior e autônoma.

Leia o `INFOGRAFICO_UNIVERSAL.md` que está no kit para entender exatamente como os dados fluem sem causar bugs.
